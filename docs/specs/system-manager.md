# System Manager spec

**Version**: v1.0 · 2026-06-24 (implemented)
**Status**: Implemented — `/system-manager` skill + `build-dashboard.py` + the `system/` data store.
**Owner**: Operator-triggered (the standing System-layer role).

## Why it exists

The architecture is constitutional three layers — **System / Tenant / Campaign** —
but only two had owners. Campaigns are owned by the Campaign Manager; tenant
learnings graduate into tenant playbooks (`graduate-then-cite`). The **System
layer** — the system's own machinery: specs, skills, agents, hooks, operator UX —
was improved ad hoc, by the operator, with no durable backlog. The System Manager
is that missing owner. It is the structural sibling of the Campaign Manager, one
layer up: CM orchestrates work *inside* campaigns; the System Manager orchestrates
improvement *of the machine that runs them*.

**Anchoring principle:** the system serves the campaigns, never the reverse. Every
backlog item must trace to a concrete pain (a failure, a friction, a named risk).
No speculative gold-plating.

## Data model

All under `system/`. The YAML is authoritative; the HTML is generated (same
data-driven contract as the campaign operator surfaces).

**Canonical store (SYS-025).** These files are git-tracked, so every `.claude/worktrees/*`
checkout holds its own copy. The canonical store is the **main checkout** (`repo_paths.data_root`).
A write made to a worktree-local copy silently forks the backlog and re-mints ids the main store
has already taken — the root cause of the 2026-06-29 SYS-018/019 collision. All reads and writes
(by the agent and by helpers) resolve through `sysdata.py`, which prints the canonical absolute
paths, allocates the next free id by scanning backlog + ideas + the append-only audit log (so a
promoted/deleted id is never reused), and guards against an existing duplicate. `build-dashboard.py`
runs the same guard on every render. Ids are unique + monotonic; legacy lettered ids (`SYS-D1/D2`)
are historical and never re-minted.

### `backlog.yaml` — tickets
```
items:
  - id: SYS-NNN                 # unique, monotonic
    title: <short handle>
    status: todo | in_progress | done | killed
    priority: P1 | P2 | P3
    raised_by: <person | diagnostic | retro>
    date: <YYYY-MM-DD>          # absolute
    source: <where it came from>
    summary: <one line, shown on the card>
    description: <full what + why>
    pr: { number: <n>, url: <url> }   # OPTIONAL — only once a change is in flight
```

### `ideas.yaml` — idea inbox (untriaged)
Same shape minus `status`/`priority`/`pr`. The inbox is a holding pen; capture is
frictionless and the operator never writes the paragraph — the System Manager
enriches a one-line capture into a stand-alone record **at capture time** (so it
triages cold weeks later, by anyone).

### `audit-log.yaml` — append-only history
```
entries:
  - date: <YYYY-MM-DD>
    ref: <SYS-NNN | IDEA-NNN>
    event: captured | triaged | started | shipped | killed | merged | retro | note
    detail: <one line>
```

### Priority
- **P1** urgent — blocks correct operation or erodes operator trust.
- **P2** debt — real friction or known gap, not blocking.
- **P3** opportunistic — nice to have, do when adjacent.

(Renders onto the existing `.prio--now/next/later` colour family: red / amber / grey.)

## The operator surface — `system/dashboard.html`

Generated by `build-dashboard.py`. A kanban:
- **To Do** with P1/P2/P3 swimlanes · **In Progress** · **Done** (recent).
- **Priority filter** chips (P1/P2/P3) — hide/show live, counts update.
- **Inbox chip** → **triage lightbox**: each idea collapses to title + summary +
  who/when, expands to the full description, with promote (P1/P2/P3) / merge / kill.
- **Audit history** below the board.
- **Run a retro** + **Capture an idea** buttons at the top.

A standalone HTML file can't call Claude, so every action button copies the exact
`/system-manager …` chat command to the clipboard and shows a toast — the read
surface hands work back to chat, the same pattern the campaign gallery uses.

> Re-render after any data write: `python .claude/skills/system-manager/build-dashboard.py`

## Workflows

See `.claude/skills/system-manager/SKILL.md` for the operating detail. Four jobs:
**Capture** (enrich → inbox) · **Triage** (promote/merge/kill → backlog) ·
**Retro** (action items → backlog, lessons → memory, record → `docs/retros/`) ·
**Groom** (reconcile diagnostics + backlog, surface one prioritisation decision).

## Boundaries
- Independent of campaigns — never touches live campaign artifacts.
- Consumes the diagnostics (`system-smoke-test`, `system-drift-watcher`,
  `nav-audit`, `cm-audit`) as sensors; it does not replace them. They detect; the
  System Manager turns findings into prioritised, actioned work.
- One decision per human moment — batch findings, auto-apply only the unambiguous,
  gate the operator on anything with judgement.
